# [MOZI] 노인 맞춤형 통합 복지 안내 서비스 PRD

## 1. 한 줄 요약

복잡한 검색과 서류 확인 없이, **대화형 챗봇과 상황별 맞춤 필터링**을 통해 노년층이 자신에게 딱 맞는 복지 혜택을 쉽게 찾고 저장할 수 있도록 돕는 **백엔드(API) 서비스**.

---

## 2. 프로젝트 컨텍스트

- **개발 규모**: 1인 개발 (백엔드 → 프론트엔드 순차 진행)
- **개발 목적**: 졸업 프로젝트 + 풀스택 1인 서비스 완성 경험
- **배포 방식**: 최종 발표 시연용 임시 배포 (AWS 예정, 실 사용자 운영 X)
- **현재 단계**: 백엔드 프로젝트 완성이 최우선 목표

---

## 3. 목표 사용자

- **주 타겟**: 디지털 기기 조작이나 복잡한 검색에 익숙하지 않은 **65세 이상 노년층**, 또는 그들을 대신해 복지 혜택을 찾는 보호자
- **겪고 있는 문제**:
  - 기존 공공 복지 사이트(예: 복지로)는 정보가 너무 방대함
  - 자신이 혜택 대상자(소득분위, 가구형태 등)에 해당하는지 스스로 판단하기 어려움
  - 검색/필터링 UI가 노년층에게 진입장벽이 높음

## 4. 핵심 기능 (MVP - Backend API 관점)

### 4-1. 인증 / 사용자 관리 API
- 회원가입, 로그인 (JWT 발급)
- 시연용 가짜 사용자 데이터로 검증

### 4-2. 맞춤형 프로필 설정 API
- 사용자의 인적/상황 정보를 입력받아 DB에 저장/수정/조회
- 입력 필드: 거주지(시도/시군구), 생년월일, 성별, 소득유형, 가구형태, 독거 여부, 장애 여부, 한부모/조손가정, 다자녀, 다문화/탈북민, 보훈대상자 등

### 4-3. 챗봇 연동 API (Core Hub)
- 사용자 메시지 + 프로필 데이터를 외부 챗봇 서버로 전달
- 챗봇 서버의 응답(추천 복지 리스트 + 답변 텍스트)을 프론트로 반환
- **본 백엔드는 LLM/RAG 자체 구동을 하지 않고 브릿지 역할만 수행**

### 4-4. 복지 정보 통합 조회 및 필터링 API
- 카테고리(의료/주거/생활비 등) 및 사용자 프로필 조건 기반 동적 필터링
- 데이터 소스: **복지로(중앙부처/지자체/민간) + 서울복지포털** 4개 출처 통합 조회
  - 출처별 데이터 스키마는 다르지만, 사용자에게는 통합된 형태로 제공
  - 출처 구분은 응답의 `welfareType` 필드로 확인 가능
- 페이지네이션 지원 (기본 size=10)

### 4-5. 관심 복지 북마크 API
- 사용자가 관심 있는 복지 정책을 저장(추가/삭제/조회)

## 5. 안 만들 것 (Out of Scope)

이 섹션은 **AI에게 "여기까지만 해"를 알려주는 가드레일**입니다.

### 5-1. 기능 범위 제외
- ❌ **복지 혜택 직접 신청 기능**: 공공기관 DB와 직접 연동 불가. 본 서비스는 **상세 안내 + 외부 신청 링크 제공**까지만 담당
- ❌ **관리자(Admin) 대시보드**: 데이터 입력/수정은 DB 직접 제어 또는 시드 스크립트로 대체
- ❌ **사용자 간 커뮤니티 기능**: 게시판/댓글/리뷰 없음. "추천"과 "검색"에만 집중
- ❌ **자체 LLM 구동**: 백엔드는 외부 챗봇 서버와 API 통신만 담당
- ❌ **푸시 알림, 이메일 발송**: MVP 범위 외
- ❌ **결제, 후원 기능**: 없음
- ❌ **모바일 앱**: 웹 백엔드만. 모바일은 PWA 또는 추후 검토

### 5-2. 운영/배포 범위 제외
- ❌ **실제 운영 배포**: 시연용 임시 배포만. 모니터링/알림/오토스케일링 등 운영 기능 X
- ❌ **CI/CD 파이프라인**: 수동 배포로 진행 (배포는 별도 학습 후)
- ❌ **다국어 지원**: 한국어 전용

## 7. 비기능 요구사항 (NFR)

### 7-1. 성능
- **일반 조회 API**: 응답 시간 500ms 이내 목표 (p95)
- **챗봇 연동 API**: 외부 서버 응답 포함 5초 이내 (timeout 8초)
- **동시 접속자**: 시연 환경 기준 최대 30명 가정 (실 운영 X)

### 7-2. 보안
- 모든 API는 JWT 기반 인증 (예외: 회원가입, 로그인, 일부 공개 조회)
- 비밀번호는 **BCrypt** 해싱 저장
- 시연용 가짜 데이터라도 보안 규칙 준수
- HTTPS는 배포 단계에서 적용 (개발 단계 HTTP 허용)

### 7-3. 사용성 (백엔드 영향)
- 노년층 타겟 → 에러 메시지는 **사용자 친화적 한글**로
  - ❌ "Internal Server Error"
  - ✅ "잠시 후 다시 시도해주세요. 문제가 계속되면 고객센터로 문의해주세요."
- 응답 페이지네이션 기본 `size=10` (긴 리스트로 인한 인지 부하 방지)

### 7-4. 유지보수성
- DDD 기반 폴더 구조로 도메인별 응집도 확보
- 글로벌 예외 처리, 통일 응답 포맷, 공통 BaseEntity 활용
- **출처가 다른 데이터는 별도 자식 엔티티로 분리** (스키마 변경에 강한 구조)

## 8. 외부 시스템 의존성

### 8-1. 챗봇 / LLM-RAG 서버
- **담당**: 별도 팀원 (현재 모델 개발 중)
- **통신 방식**: REST API (HTTPS)
- **백엔드의 역할**: 사용자 메시지 + 프로필 정보 → 챗봇 서버로 전달 → 응답 가공 후 반환

#### 인터페이스 설계 (Step 7 본격 매핑 적용, 2026-05-17)

> 자세한 wire format은 `docs/CHATBOT_API_CONTRACT.md` §2 참조. 변경 요약은 `docs/CHATBOT_MIGRATION_NOTES.md`.

```
POST {CHATBOT_SERVER_URL}/chat
Request:
{
  "message": "병원비 도움이 필요해",
  "conversationId": "uuid (대화 세션 식별)",
  "user": {
    "userId": 1,
    "profile": {
      "age": 78,
      "gender": "F",
      "sidoName": "서울특별시",
      "sigunguName": "강남구",
      "incomeType": "기초연금수급자",       // 한글 라벨 (Enum.label())
      "householdType": "혼자 살아요 (독거)",  // 한글 라벨
      "isDisabled": false,
      "isMultiChild": false,
      "isMulticulturalNorthDefector": false,
      "isVeteran": false
    }
  }
}

Response:
{
  "reply": "안녕하세요. 다음 복지 혜택을 추천드려요...",
  "recommendedWelfareIds": ["WLF00005442", "WLF00003965", "SEL00000001", ...],
  "conversationId": "uuid"
}
```

> 💡 추천 복지 ID는 **4개 출처 중 어느 것이든 가능** (`WLF...`는 복지로, `BOK...`는 민간, `SEL...`은 서울)

#### 사용자 프로필 전달 방식 (협의 사항)
- **옵션 A**: 매 요청마다 프로필을 함께 전송 (단순, 무상태)
- **옵션 B**: 첫 대화 시 한 번 전송 후 챗봇 서버가 캐시 (효율, 상태 관리 필요)
- **현재 채택 잠정안**: **옵션 A** (백엔드가 무상태로 동작 가능, 구현 단순)
- 챗봇 팀과 최종 결정 후 EXECUTION_PLAN Phase 5에서 확정

#### 챗봇 서버 부재 시 대응
- 개발 초기에는 **Mock 챗봇 서버**로 진행
- `application-mock.yml` 프로파일 활성화 시 더미 응답 반환
- 실 서버 연동은 챗봇 팀 준비 완료 후 통합

### 8-2. 복지 데이터 소스 ⭐
- **이미 확보된 데이터**:
  - **복지로** 사이트 크롤링 → 중앙부처 / 지자체 / 민간 복지 정보 (3종)
  - **서울복지포털** 사이트 크롤링 → 서울시 복지 정보 (1종, 별도 출처)
- 합계 **4개 출처**, 각 출처별로 데이터 스키마가 다름
- 노인 관련 필터링 완료 (전체 데이터 아닌 노인 대상으로 필터링된 상태)
- **DB 적재 방식**:
  - 출처별로 별도 자식 엔티티에 적재 (`WelfareCentral`, `WelfareLocal`, `WelfarePrivate`, `WelfareSeoul`)
  - 출처별 컬럼명은 시드 데이터 사전 정규화로 통일됨 (단일 시드 어댑터로 처리)
- 공공데이터 API 직접 연동: 본 MVP에서는 진행하지 않음 (크롤링 데이터로 충분)

### 8-3. 사용자 데이터
- 실제 사용자 없음
- **시연용 가짜 사용자 N명**(예: 5~10명)을 시드 스크립트로 생성
- 각 가짜 사용자는 다양한 프로필 조건(연령/지역/소득/가구형태)을 커버하여 시연 시나리오 다양화

## 9. 주요 사용자 시나리오

각 시나리오는 USER_FLOW.md에서 더 상세히 다룸.

### 시나리오 A. 챗봇 상담 → 추천
1. 사용자가 로그인하여 챗봇 페이지 진입
2. "병원비 도움이 필요해"라고 입력
3. 백엔드: 사용자 프로필 조회 → 챗봇 서버에 메시지+프로필 전달
4. 챗봇 서버: 분석 후 추천 복지 ID 리스트 + 답변 텍스트 반환
5. 백엔드: 추천 복지 ID로 상세 정보 조회 (4개 출처 중 해당하는 자식 엔티티에서) 후 프론트에 통합 응답

### 시나리오 B. 카테고리 기반 탐색
1. 사용자가 6대 카테고리 중 "의료" 클릭
2. 백엔드: 카테고리 + 사용자 프로필로 필터링된 복지 리스트 반환 (4개 출처 통합)

### 시나리오 C. 맞춤 정보 입력
1. 마이페이지에서 토글 UI로 자신의 상황 선택
2. 백엔드: UserProfile 업데이트
3. 다음 조회/추천부터 새 프로필 반영

### 시나리오 D. 상세 확인 + 북마크
1. 추천 카드 클릭 → 상세 페이지 진입
2. 지원 대상/금액/신청 방법 확인 (출처별로 다른 필드 표시)
3. "저장하기" 클릭 → 북마크 등록
4. 마이페이지에서 북마크 목록 조회 가능

## 10. 성공 기준 (졸업 프로젝트 관점)

- ✅ 5개 핵심 API 그룹(인증/프로필/챗봇/복지조회/북마크) 모두 정상 동작
- ✅ Postman 또는 Swagger UI로 시연 가능
- ✅ 시드 데이터 기반으로 가짜 사용자 + **4개 출처 복지 데이터**로 검색/추천이 의미 있게 동작
- ✅ 코드 테스트 커버리지 Service 레이어 70% 이상
- ✅ AWS 임시 배포 성공 (백엔드만이라도 EC2에서 외부 접근 가능)
- ✅ 발표 시연 시 챗봇 서버와 정상 연동 (또는 Mock으로라도 흐름 시연)
