## 1. 시스템 아키텍처

<p align="center">
  <img src="architecture.png" alt="AWS 아키텍처" width="90%"/>
</p>

MOZI는 AWS 기반 클라우드 인프라 위에 구축되었으며, 보안을 위해 Public / Private Subnet을 계층적으로 분리했습니다.

### 요청 흐름

```
USER
 │  ① 웹 화면 접속
 ▼
AWS Amplify ──▶ IGW ──▶ ALB (Public Subnet)
                          │  ② 백엔드 전달
                          ▼
                        EC2 (Public Subnet)
                          │
             ┌────────────┴────────────┐
             ▼                         ▼
      RDS / MySQL              FastAPI (via Cloudflare)
   ③ 사용자 정보 조회            │  ④ API 요청 처리
   ⑥ 복지 데이터 조회            ▼
                            OpenAI
                          ⑤ 답변 생성
```

### 구성 요소

| 계층 | 구성 | 설명 |
|------|------|------|
| **Frontend** | AWS Amplify | 웹 화면 호스팅 및 배포 |
| **네트워크** | IGW, ALB, Cloudflare | 외부 HTTPS 접속 및 도메인 연결, 트래픽 분산 |
| **Backend** | EC2, FastAPI | API 요청 처리 및 RAG 파이프라인 실행 |
| **Database** | RDS (MySQL) | 사용자 정보 및 복지 데이터 저장 |
| **AI** | OpenAI | 임베딩 및 답변 생성 |

### 네트워크 & 보안

- **VPC** `172.31.0.0/16` 내 다중 가용 영역(AZ) 구성 — `ap-northeast-2a` / `2c` / `2b`
- **Public Subnet** — ALB, EC2 배치 / **Private Subnet** — RDS 배치로 DB 외부 노출 차단
- 보안 그룹으로 계층별 접근 제어: `mozi-alb-sg`, `mozi-ec2-sg`, `mozi-rds-sg`

## 2. 데이터 구성 & 파이프라인

### 수집 데이터

| 항목 | 내용 |
|------|------|
| **수집처** | 복지로, 서울복지포털 |
| **수집 규모** | 총 **1,473건** (중앙부처 87 / 지자체 1,324 / 민간 35 / 서울시 37) |
| **주요 항목** | 정책명·요약, 담당부서·문의처, 지원대상·가구상황, 서비스내용·신청방법, 분류코드 |
| **수집 기술** | 웹크롤링 (Python, requests) |
| **수집 주기** | 주 1회 (월요일 03:00–04:00) |

### 파이프라인 (5단계)

```
1. 수집        →  2. 정제         →  3. 적재       →  4. 임베딩              →  5. 인덱싱
Python/requests   Pandas            MySQL            text-embedding-3-large    FAISS / BM25
크롤링            필드 통일·중복제거   정형 DB 저장     복지 문서 → 벡터 변환      벡터/키워드 인덱스 구축
```

### 데이터 품질 계층 (Medallion Architecture)

데이터 엔지니어링의 메달리온 아키텍처를 RAG에 적용했습니다.

| Bronze (Raw) | Silver (정제) | Gold (최종) |
|--------------|---------------|-------------|
| 원본 그대로 | 유형 구분, 텍스트 추출 | 청킹 완료 |
| PDF, HWP, JSON, CSV, HTML | 인코딩·포맷 통일, 기본 전처리 | 메타데이터 정규화, 중복 제거, 벡터DB 적재 |

**최종 산출물:** MySQL 복지 데이터 + FAISS/BM25 검색 인덱스 → RAG 기반 복지 추천·답변에 활용

## 3. RAG 파이프라인

MOZI의 핵심은 **범용 LLM이 임의로 답하는 것이 아니라, 실제 복지 데이터에 근거해 답변**하도록 RAG를 구성하고, 이를 정량 지표로 검증하며 단계적으로 고도화한 과정입니다.

### 3.1 개선 서사 (Baseline → 최종)

| 단계 | 적용 기법 | 성과 |
|------|-----------|------|
| **Baseline** | Semantic 검색만 | Hit@3 `0.4624` / Faithfulness `0.75` |
| **+ Hybrid Search** | BM25 + Semantic (RRF 결합) | Hit@3 `0.6022` |
| **+ Reranker** | `bge-v2-m3`로 후보 문서 재정렬 | Hit@3 `0.7849` |
| **+ Prompt Engineering** | 금지 규칙 + Few-shot 예시 | Faithfulness `0.8741` |
| **+ 예외 처리 / Multi-turn** | Intent 게이트 + Rolling Summary | 불필요한 검색·LLM 호출 및 오응답 감소 |

**최종 성능:** Answer Correctness `0.5335 → 0.5571`, Faithfulness `0.7499 → 0.8741`

### 3.2 단계별 상세

**① Chunking** — 복지 데이터는 레코드(record) 단위 청킹을 채택했습니다. 레코드 길이 분포상 중앙값 593자, P90 991자였으며, 실험 결과 512~1024 토큰 구간에서 가장 안정적이었고 nDCG@k 기준 row 단위 청킹이 512/1024 토큰 청킹보다 전반적으로 우수했습니다.

**② Embedding** — `text-embedding-3-large`를 선정했습니다. (MTEB 64.6% / MIRACL 다국어 검색 54.9%, 3072-dim) 다국어 검색 성능이 특히 우수해 한국어 복지 데이터에 적합합니다.

**③ Vector Store** — 대규모·성능 중심 검색을 위해 **FAISS**를 채택했습니다. (GPU 가속 지원, HNSW 인덱싱, 대규모 데이터셋 처리 속도 우수)

**④ Hybrid Search** — 시맨틱 검색(의미 기반)만으로는 "성북구"처럼 사용자 조건과 유사한 "강북구"·"성동구" 등 비슷한 지명의 복지가 상위에 배치되어, 정답 문서가 낮은 순위로 밀리는 문제가 있었습니다. 이를 해결하기 위해 **BM25 키워드 검색을 추가**해 핵심 조건 매칭을 보완하고, RRF(Reciprocal Rank Fusion)로 두 검색 결과를 병합했습니다.
- 최종 설정: `DENSE_K=50`, `BM25_K=50`, `TOP_N=50`

**⑤ Reranker** — Hybrid Search로 검색된 후보 문서를 재정렬합니다. Cohere-v3.5가 최고 성능이었으나 응답 시간이 `bge-v2-m3` 대비 약 3배 길어, 로컬 모델로 비용 부담이 없고 성능 차이도 크지 않은 **`bge-v2-m3`**를 최종 선정했습니다.

**⑥ 예외 처리 (Intent Gate)** — 검색 전에 입력 의도를 분류하여 복지 관련 질문만 RAG 파이프라인으로 전달합니다.

| 분류 | 입력 예시 | 처리 방식 |
|------|-----------|-----------|
| `WELFARE` | "노인 일자리 알려줘" | 기존 RAG 파이프라인 수행 |
| `CHITCHAT` | "안녕", "고마워" | 복지 안내형 응답 반환 |
| `OUT_OF_SCOPE` | "오늘 날씨 어때?" | 범위 밖 질문 안내 응답 반환 |
| `HARMFUL` | 욕설, 혐오·차별 표현 | 부적절 표현 안내 및 복지 질문 유도 |

→ 복지 무관 입력은 검색·리랭킹·LLM 생성을 모두 스킵해 불필요한 비용과 응답 지연을 줄입니다.

**⑦ Multi-turn** — "요약 + 직전 1턴 원문" 하이브리드 메모리 구조를 사용합니다.
- **Query Rewriting** — "그건 어떻게 신청해?" → "기초연금은 어떻게 신청하나요?" (지시어를 실제 단어로 치환)
- **Followup 감지** — 재작성으로 문장이 바뀌면 후속질의로 판정 → 묻는 것만 답변 + 의도분류 스킵
- **Rolling Summary** — 매 턴 답변 후 직전 Q&A를 요약에 흡수 (복지명·금액·전화번호 등 핵심 정보 보존 규칙 적용) → 자격 판단 일관성 유지, 컨텍스트 무한 증가 방지

### 3.3 모델 & 비용

- **메인 답변**: `gpt-4.1-mini` / **보조 작업(재작성·요약·분류)**: `gpt-4.1-nano` → 비용 최적화
- System/User 분리로 prompt caching 적용
- LangSmith로 계층별 트레이스 관리
