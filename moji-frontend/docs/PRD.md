# [MOZI Frontend] 노인 맞춤형 통합 복지 안내 — 프론트엔드 PRD

## 1. 한 줄 요약

복잡한 검색·서류 확인 없이, **대화형 챗봇과 상황별 맞춤 필터**로 노년층이 자신에게 맞는 복지 혜택을 쉽게 찾고 저장할 수 있는 **웹 프론트엔드(SPA)**.
백엔드는 이미 완성된 16개 REST API를 제공한다 → 프론트는 이를 충실히 소비해 **노년층이 직관적으로 쓸 수 있는 UI**를 만든다.

---

## 2. 프로젝트 컨텍스트

- **개발 규모**: 1인 개발 (백엔드 완료 → 프론트엔드 시작 단계)
- **개발 목적**: 졸업 프로젝트 + 풀스택 1인 서비스 완성 경험
- **현재 단계**: 프론트엔드 Phase 0 (docs 작성 중)
- **배포 방식**: 시연용 임시 배포 (AWS S3 + CloudFront 또는 Amplify 검토, Phase 7)
- **개발자 숙련도(중요)**: 프론트엔드 거의 처음. React/Vite/Tailwind/fetch/react-router 학습 병행
  - → UI 복잡도와 외부 의존성을 **의도적으로 낮게 유지**한다 (TypeScript X, 캐싱 라이브러리 X, SSR X)

---

## 3. 목표 사용자

- **주 타겟**: 디지털 기기 조작이 익숙하지 않은 **65세 이상 노년층**, 또는 그들 대신 복지를 찾는 보호자
- **사용 시 겪는 어려움 (UI 관점)**:
  - 작은 폰트·작은 버튼·복잡한 메뉴를 보면 첫 화면에서 이탈
  - 양식 입력 시 무엇을 적어야 할지 불명확하면 포기
  - "오류" 같은 기술 용어 메시지를 보면 작업을 중단
  - 여러 단계로 나뉜 흐름은 어디까지 왔는지 인지 어려움
- **UI 대응 원칙** (자세히 §7):
  - 폰트·버튼은 시각적으로 크게
  - 입력란마다 라벨 + 예시
  - 에러는 "사용자가 무엇을 하면 되는지" 친근한 한글로
  - 한 화면 한 작업 (멀티 액션 회피)

## 4. 핵심 화면 (MVP)

> 백엔드의 핵심 기능 5개 + 노년층 UX 고려 → 프론트 화면은 다음 9개로 구성한다.
> 화면별 상세 명세는 `SCREEN_SPEC.md` 참조.

### 4-1. 로그인 (`/login`)
- 이메일·비밀번호 입력 → JWT 발급 후 메모리/sessionStorage 저장
- 비로그인 사용자가 보호된 페이지에 접근하려 하면 이리로 리다이렉트

### 4-2. 회원가입 (`/signup`)
- 이메일·비밀번호 2회 입력 → 가입 즉시 토큰 발급되어 자동 로그인 상태로 진입
- 가입 직후 **프로필 입력 권유 모달**을 띄워 챗봇 추천 정확도를 높이도록 유도(건너뛰기 가능)

### 4-3. 챗봇 메인 / 홈 (`/`)
- **본 서비스의 Core 페이지**
- 큰 입력란 + 전송 버튼
- 챗봇 답변 텍스트 + 추천 복지 카드(있을 때만) 노출
- "새 대화 시작" 버튼으로 conversationId 초기화
- 비로그인 시 진입 차단 → 로그인 페이지로

### 4-4. 복지 검색 (`/welfares`)
- 키워드·카테고리·지역·출처 필터로 동적 검색
- 페이지 번호 페이지네이션 (이전/다음 + 최대 5개 번호 chip, 무한 스크롤 X — 노년층 인지 부하)
- 결과 카드 리스트 (출처 뱃지 + 제목 + 요약 + 북마크 토글)
- **비로그인도 접근 가능** (optional auth) — 단, 본인 조건 자동 적용은 비활성

### 4-5. 복지 상세 (`/welfares/:id`)
- 카드 클릭 시 진입. 출처별로 다른 필드 표시 (CENTRAL/LOCAL/PRIVATE/SEOUL)
- 지원 대상, 신청 방법, 담당 기관, 외부 신청 링크
- 큰 "저장하기" 버튼 (북마크 토글)
- 비로그인도 접근 가능 (북마크 버튼만 비활성)

### 4-6. 북마크 목록 (`/bookmarks`)
- 내가 저장한 복지 카드 리스트 (최신순)
- 페이지 번호 페이지네이션 (검색 페이지와 동일 컴포넌트)
- 빈 상태: "아직 저장한 복지가 없어요" + 검색 페이지 안내

### 4-7. 마이페이지 (`/me`)
- 진입 시 프로필 정보 조회 (isCompleted=false면 입력 권유 카드)
- 하위 액션 링크: "프로필 수정 / 비밀번호 변경 / 회원 탈퇴"

### 4-8. 프로필 수정 (`/me/profile`)
- 생년월일·성별·시도/시군구·소득유형·가구형태·6개 boolean 토글
- 부분 수정 가능(PUT). 미입력 필드는 변경 없음
- 안내 문구: "입력하실수록 추천이 더 정확해집니다"

### 4-9. 비밀번호 변경 (`/me/password`)
- 현재 비밀번호 + 새 비밀번호 + 새 비밀번호 확인
- 성공 시 안내 토스트 + 자동 재로그인 필요 안내
  - (백엔드가 모든 refreshToken을 무효화하므로 다른 기기는 자동 로그아웃됨)

> 회원 탈퇴는 별도 페이지가 아닌 마이페이지 내 확인 모달로 처리.

## 5. 화면별 사용자 가치

> 각 화면이 사용자에게 어떤 가치를 주는가. 화면을 더할지/뺄지 판단할 때 기준.

| 화면 | 사용자가 얻는 가치 |
|---|---|
| 챗봇 메인 | "내 상황을 자연어로 말하면 알아서 추천해 줘" — 검색 진입장벽 0 |
| 복지 검색 | "원하는 키워드/지역으로 직접 탐색" — 챗봇 외 통제권 |
| 복지 상세 | "지원 대상에 내가 해당되는지, 어떻게 신청하는지 확인" — 의사결정 |
| 북마크 | "관심 복지를 모아두기" — 가족과 공유·재방문 |
| 프로필 | "한 번 입력하면 모든 추천이 자동 맞춤" — 반복 입력 회피 |
| 로그인/가입 | "내 정보를 안전하게 보관 + 다음 방문 시 이어 사용" |

## 6. 안 만들 것 (Out of Scope)

> AI에게 "여기까지만 해"를 알려주는 가드레일. 백엔드 PRD §5와 동기화.

### 6-1. 기능 범위 제외
- ❌ **복지 직접 신청 화면** — 외부 신청 링크(`detailUrl`)로 이동시키는 것만
- ❌ **관리자 화면 / 데이터 관리 UI** — 백엔드 시드/DB로 관리
- ❌ **사용자 간 커뮤니티 / 댓글 / 후기** — 추천·검색에만 집중
- ❌ **챗봇 자체 LLM 호출** — 백엔드 `/api/chat` 브릿지만 사용
- ❌ **푸시 알림 / 이메일 발송 화면** — MVP 범위 외
- ❌ **결제·후원 화면** — 없음
- ❌ **다국어 토글** — 한국어 전용
- ❌ **다크모드 전환** — 라이트 모드만 (선택 사항으로도 추가 안 함)
- ❌ **PWA 설치 배너 / 오프라인 캐시** — MVP 범위 외

### 6-2. 기술/구조 제외
- ❌ **TypeScript** — JavaScript 확정 (학습 곡선 관리)
- ❌ **SSR / SSG (Next.js 등)** — Vite + CSR 단일 페이지
- ❌ **상태 캐싱 라이브러리 (tanstack-query, SWR)** — 본 규모에 과함
- ❌ **CSS-in-JS (styled-components, emotion)** — Tailwind 통일
- ❌ **자체 디자인 시스템 / Storybook** — 졸업 프로젝트 범위 외
- ❌ **E2E 자동화 테스트 (Playwright/Cypress)** — 수동 검증으로 대체

### 6-3. 운영 범위 제외
- ❌ **실 운영 배포** — 시연용 임시 배포만
- ❌ **모니터링 / 에러 트래킹 (Sentry 등)** — 콘솔 로그로 충분
- ❌ **CI/CD 파이프라인** — 수동 배포

## 7. 비기능 요구사항 (NFR)

### 7-1. 노년층 UX (이 프로젝트의 차별화 핵심) ⭐
모든 NFR 중 **가장 중요**. 자세한 토큰·규칙은 `STYLING_GUIDE.md` 참조.

| 영역 | 기준 |
|---|---|
| 본문 폰트 크기 | 최소 18px (`text-senior-base`) — 일반 웹의 14~16px보다 큼 |
| 강조 폰트 | 22~28px (`text-senior-lg` / `text-senior-xl`) |
| 인터랙티브 요소 높이 | 최소 48px (`h-12`) — 손가락·노안 친화 |
| 색 대비 | WCAG AA 4.5:1 이상 |
| 라벨 | 모든 입력란에 명시적 라벨 (placeholder만 X) |
| 에러 메시지 | "사용자가 무엇을 하면 되는지" 한글로. 백엔드 errorCode별 사용자 친화 메시지를 그대로 사용 |
| 토스트 표시 시간 | 4초 이상 |
| 페이지네이션 | 페이지 번호 방식 — 이전/다음 + 최대 5개 번호 chip (무한 스크롤 X). 모든 버튼 h-12(48px) 이상 |
| 멀티 액션 회피 | 한 화면당 핵심 액션 1~2개. 부가 액션은 보조적으로 |

### 7-2. 성능 (체감 위주)
- 첫 화면 진입(로그인 후 챗봇): **3초 이내** 인터랙션 가능
- 검색·북마크: 백엔드 응답 + 렌더 합쳐 **1초 이내** 목표
- 챗봇 응답: 백엔드 8초 타임아웃에 맞춰 **로딩 인디케이터 필수**
- 동시 접속자: 시연 시 최대 30명 가정 (성능 튜닝은 본격 도입 X)

### 7-3. 접근성 (a11y)
- 키보드만으로 모든 액션 가능 (Tab 이동, Enter 제출)
- 시맨틱 HTML 우선 (`<button>`, `<label>`, `<nav>` 등)
- `aria-*` 속성은 필요한 곳에만 (스크린리더 안내가 부족할 때)
- 노년층은 스크린리더 사용 비중이 낮지만, 기본 시맨틱은 챙긴다

### 7-4. 반응형
- **모바일 우선**: 노년층 시연 시 태블릿/대화면 스마트폰 사용 빈도 높음
- 브레이크포인트: Tailwind 기본(`sm 640 / md 768 / lg 1024`)
- 본 MVP에선 PC와 모바일 모두 자연스럽게 동작하면 OK (별도 모바일 앱 X)

### 7-5. 보안 (프론트 책임 범위)
- 토큰 저장 정책 엄수 (CLAUDE.md §4-5): accessToken=메모리, refreshToken=sessionStorage
- HTTPS 적용은 배포 단계에서 (개발 단계 HTTP 허용)
- XSS 방지를 위해 사용자 입력을 그대로 `dangerouslySetInnerHTML`에 넣지 않는다
- 백엔드 응답이 사용자에게 안전한 한글 메시지를 주므로, 프론트에서 임의 가공·노출 추가 금지

## 8. 외부 시스템 의존성

### 8-1. 백엔드 API 서버
- **위치(개발)**: `http://localhost:8080/api`
- **위치(배포 예정)**: AWS 배포 시 결정 (Phase 7)
- **명세**: `../../backend_project/docs/API_SPEC.md` (16개 엔드포인트)
- **인증**: JWT Bearer (Access 1시간, Refresh 7일 Rotation)
- **응답 포맷**: `ApiResponse<T>` 통일 래퍼 (`status`/`message`/`data` 또는 `errorCode`/`fields`/`timestamp`)
- **프론트 책임**:
  - Authorization 헤더 자동 부착
  - `TOKEN_EXPIRED` 자동 재시도 (refresh → 원 요청 재호출)
  - `VALIDATION_FAILED` 시 `fields` 매핑
  - `errorCode`별 사용자 친화 처리

### 8-2. 챗봇 서버
- **프론트는 직접 호출하지 않는다.** 반드시 백엔드 `/api/chat` 경유.
- 챗봇 서버는 별도 팀이 LLM + RAG로 개발 중
- 백엔드가 Mock 모드를 지원하므로, 챗봇 서버 부재 시에도 프론트 시연 가능

## 10. 주요 사용자 시나리오 (요약)

> 상세 mermaid는 `USER_FLOW.md` 참조.

### 시나리오 A. 첫 방문 → 챗봇 추천
1. `/signup`에서 가입 → 자동 로그인 + 프로필 입력 권유 모달
2. 프로필 일부 입력 후 닫기 → `/` (챗봇 메인) 진입
3. "병원비 도움이 필요해" 입력 → 백엔드 `/api/chat` 호출
4. 답변 텍스트 + 추천 복지 카드 표시 → 카드 클릭 시 상세 페이지

### 시나리오 B. 재방문 (토큰 자동 복구)
1. 브라우저 새로고침 → accessToken은 메모리에서 소실
2. sessionStorage의 refreshToken 발견 → `/api/auth/refresh` 호출
3. 새 토큰 쌍 받음 → 원래 페이지에 머무름

### 시나리오 C. 직접 검색 → 북마크
1. `/welfares`에서 "노인 일자리" 키워드 + "내 조건 자동 적용" 토글
2. 결과 카드 리스트 → 카드 클릭 → 상세 페이지
3. "저장하기" 클릭 → POST `/api/bookmarks/{welfareId}`
4. `/bookmarks`에서 모아 보기

### 시나리오 D. 비밀번호 변경
1. `/me/password`에서 현재·새 비밀번호 입력
2. 성공 토스트 + "다른 기기에서는 다시 로그인이 필요해요" 안내
3. 본 기기는 새 토큰 쌍을 자동으로 받아 그대로 사용 가능

## 11. 성공 기준 (졸업 프로젝트 관점)
